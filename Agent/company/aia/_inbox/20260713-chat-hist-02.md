### Response Summary:

**Q1: Cost ต่ำสุด 5 ชม:**

```
Medium App (smallest available, default):
├── 0.5 DBU/hour
├── 5 hours × 0.5 = 2.5 DBUs total

Azure Premium tier: ~$0.55/DBU (list price)
Base cost: 2.5 × $0.55 = ~$1.38

With AIA enterprise discount (~20-30%):
~$0.97-$1.10 approximate

Large App: 1.0 DBU/hour × 5 × $0.55 = ~$2.75
```

**Sizes comparison:**
| Size | DBU/hr | 5 hrs cost |
|---|---|---|
| Medium ⭐ | 0.5 | ~$1.38 |
| Large | 1.0 | ~$2.75 |

**Q2: Stop/Terminate/Delete + Idle billing:**

**3 Actions:**
1. **STOP (Suspend)** — App exists, config kept, compute stops, restart anytime
2. **DELETE** — App removed, config gone, need redeploy
3. **UPDATE (Redeploy)** — Update code/config, restart

**Methods:**
- UI: Apps page → app details → Stop/Delete buttons
- CLI: `databricks apps stop <name>` / `delete` / `start`
- REST API: `POST /api/2.0/apps/<name>/stop`
- Python SDK: `w.apps.stop(name)` / `.delete()` / `.start()`

**Cost when STOPPED: $0** ✅

```
App STATUS = STOPPED:
├── Compute suspended
├── ⭐ No DBU consumption
├── ⭐ No hourly billing
├── Config + code stored (free)
└── Restart takes ~10-60 seconds

App STATUS = RUNNING:
├── Compute active
├── DBU/hour billing
├── Bills per second
└── Even if no users → still billing
```

**Cost Optimization Strategy provided:**

Realistic monthly cost table:
| Configuration | Hrs/Month | Medium | Large |
|---|---|---|---|
| 5 hrs (one-off) | 5 | $1.38 | $2.75 |
| 5 hrs/day × 22 workdays | 110 | $30.25 | $60.50 |
| Business hours (9-18) | 220 | $60.50 | $121 |
| Full 24/7 | 720 | $198 | $396 |

**Recommendation:**
- PoC: Test 5-10 hrs first (~$1.50-5)
- Production: Business hours automation ~$60-95/month
- vs 24/7: $198/month
- Savings: ~70% via smart start/stop

## Turn 16: Cross-Workspace + Row-Level Access

### Sin's Question:
> อีกอย่างคือ ทีนี้เราจะให้ user คนล่ะ ws คนล่ะ env เข้ามา access ได้มั้ย โดยไม่ต้องเข้ามาใน ws เรา 
> แล้วจะ control row level access ได้มั้ยนะ

### Response Summary:

**Q1: Cross-workspace/cross-env access — YES**

Concept: User access = Account tier, NOT Workspace tier

```
Databricks Account (AIA):
├── Account Users (from various workspaces)
├── ⭐ Everyone is account member
└── ⭐ But in different workspaces

Databricks App:
├── Deploy in DEV IT workspace
├── Sharing = CAN_USE at Account level
├── ⭐ Access via URL, not workspace navigation
└── ⭐ Cross-env dissolved at account tier
```

From official docs:
> "Users don't need to belong to the same workspace, but they must be part of the same Databricks account."

**Client Access Flow:**
1. Sin provides URL
2. Client opens URL in browser
3. Databricks SSO authenticates
4. Client identified as account user (not workspace member)
5. Client interacts with app (never enters DEV IT workspace)

**Prerequisites:**
- Same Databricks Account
- Client users registered as Account Users
- App shared with correct groups
- SSO working

**Q2: Row-Level Access Control — YES via User Authorization**

**2 Options:**

**Option A: App Authorization (Service Principal)**
- App has SP identity
- Fixed permissions
- ⚠️ All users share same permissions
- ⚠️ Cannot filter per user (natively)
- Not suitable for chargeback

**Option B: User Authorization (Public Preview)** ⭐
- App forwards user's OAuth token
- Queries run as user's identity
- UC RLS applies automatically
- ✅ Per-user filtering
- ✅ Chargeback enforced by UC

From docs:
> "User authorization enables fine-grained access control by applying Unity Catalog features like row-level filters and column masks to app activity."

**How User Authorization Works:**
```
Client login → Databricks account authenticates
   ↓
App receives request with headers:
   X-Forwarded-User: john@aia.com
   X-Forwarded-Access-Token: <john's OAuth token>
   ↓
App uses john's token to query
   ↓
UC evaluates:
   ├── john has SELECT? ✓
   ├── Apply row filter
   ├── Check group membership
   └── Filter rows automatically
   ↓
Return john's team rows only
```

**Sin's implementation pattern:**
```python
user_email = st.context.headers.get("X-Forwarded-Email")
user_token = st.context.headers.get("X-Forwarded-Access-Token")

def get_sql_connection():
    cfg = Config(token=user_token)  # ⭐ user's token
    return sql.connect(...)
```

**UC Row Filter:**
```sql
CREATE OR REPLACE FUNCTION cost_platform.gold.team_row_filter(team STRING)
RETURN
    is_account_group_member('de-team')
    OR is_account_group_member(concat('client-', lower(team)));

ALTER TABLE cost_platform.gold.cost_wide
    SET ROW FILTER cost_platform.gold.team_row_filter ON (tag_team);
```

**Verification Checklist provided:**
- Same Databricks Account
- User Authorization enabled (workspace admin)
- OAuth scope allowlist
- Client users registered
- Client account groups exist
- Cost catalog NOT workspace-bound

**Watch-outs identified:**
- User Authorization = Public Preview status
- Cost Catalog Workspace-Bound possibility
- Different Databricks Accounts (would break)
- OAuth Scope Restriction

**Test Plan provided:**
- Phase 1: Prerequisites check
- Phase 2: PoC small scale
- Phase 3: Cross-Env Verification

## Turn 18: Az Login SSL Certificate Error

### Sin's Question with Screenshot:
> Az login ไม่ได้ มีerror นี้อ่ะ ผมต้อง setup cert อะไรรึป่าวนะ

Screenshot showed:
```
$ az login
HTTPSConnectionPool(host='login.microsoftonline.com', port=443): 
Max retries exceeded with url: /organizations/v2.0/.well-known/openid-configuration 
(Caused by SSLError(SSLCertVerificationError(1, 
'[SSL: CERTIFICATE_VERIFY_FAILED] certificate verify failed: 
unable to get local issuer certificate (_ssl.c:1032)')))

Certificate verification failed. This typically happens when using Azure CLI 
behind a proxy that intercepts traffic with a self-signed certificate. 
Please add this certificate to the trusted CA bundle.
```

### Response Summary:

**Root cause:** AIA's corporate proxy (Zscaler / Netskope) intercepts HTTPS and re-signs with own CA. Python (Azure CLI) doesn't trust that CA.

**Fix: Configure Corporate CA Bundle**

**Step 1: Get AIA's Corporate CA Cert**
- IT ticket for corporate root CA
- Or export from Windows Cert Store (certmgr.msc)
- Trusted Root Certification Authorities → find AIA/Zscaler cert
- Right-click → All Tasks → Export → Base-64 encoded X.509 (.CER)

**Step 2: Setup env var**
```bash
export REQUESTS_CA_BUNDLE="/c/path/to/aia-corporate-ca.crt"
az login --tenant <AIA-tenant>
```

**Alternative methods provided:**
- Persistent config (Windows environment variable)
- Add to Python's cert bundle (append to certifi cacert.pem)
- Azure CLI specific config

**Do NOT do:** Disable verification (`AZURE_CLI_DISABLE_CONNECTION_VERIFICATION=1`) — security risk

**Related tools affected:**
- pip: `PIP_CERT`
- git: `http.sslCAInfo`
- curl/wget: `CURL_CA_BUNDLE`
- npm: `cafile`
- databricks CLI: same `REQUESTS_CA_BUNDLE`

**Efficient one-time setup:**
```bash
# ~/.bashrc
export REQUESTS_CA_BUNDLE="/c/certs/aia-corporate-ca.crt"
export PIP_CERT="$REQUESTS_CA_BUNDLE"
export CURL_CA_BUNDLE="$REQUESTS_CA_BUNDLE"
export NODE_EXTRA_CA_CERTS="$REQUESTS_CA_BUNDLE"
export SSL_CERT_FILE="$REQUESTS_CA_BUNDLE"
```
