## 3. Recommendation

### ⭐ Primary: **Option D+**
> *"Grant the data. Hand over the dashboard file. They own the object."*

**Sin's team (all inside coredata — zero deployment into their workspace):**
1. Make sure the catalog is reachable from departmental (`OPEN`, or add them with `BINDING_TYPE_READ_ONLY`).
2. `GRANT USE CATALOG / USE SCHEMA / SELECT` on the gold table + **`GRANT EXECUTE`** on the filter function → to their **ACCOUNT groups**.
3. Attach the **row filter** (or an **ABAC policy** — Databricks now recommends ABAC over per-table `SET ROW FILTER`).
4. Build the dashboard in coredata → **Export** `.lvdash.json`.
5. Hand over the file. Done.

**Departmental admin (one time, ~15 min):**
6. **Import** the `.lvdash.json` → move it to `/Workspace/Shared/…`.
7. Point the dataset at **their** SQL warehouse.
8. **Publish → "Individual data permissions"** 🚨 *non-negotiable — this is what makes C5 real.*
9. Share to the team groups (named groups only — **never** "Anyone in my account can view").
10. Add a **monthly schedule** + subscribers.

**Why this and nothing else:** it is the only shape where the artifact is genuinely **inside their workspace** (C6) **and** **their warehouse pays** (C7) **and** **RLS is real** (C5) — because all three collapse onto one fact: **the publisher's identity and warehouse run the query.** Sin's team never touches their workspace — a `.lvdash.json` is a **file**, not a deployment (C4 holds cleanly). Everything is **GA**.

### Runner-up / interim: **Option C**
Zero effort from the departmental side; Sin can ship it alone this week. **But** fails C6 + C7, and **breaks silently if the catalog is `ISOLATED`**.
→ **Best politics: ship C to unblock now, hand over the `.lvdash.json` for D+ this month.**

### Fallbacks
- **Option O (push model)** — if they refuse to let us grant on our catalog: IT writes per-team data into a **departmental-owned** catalog; *they* grant their own groups. Strictly easier to sell; costs a data copy.
- **Option A′** (row filter keyed on `current_user()` + an IT-maintained mapping table) — **needs nobody's permission**; Sin can demo end-to-end today while the group question grinds through the org. Accept the joiner/mover/leaver debt; design the mapping table with **both** `account_group` and `user_email` columns so migrating back is a data change, not a schema change.
- **Option N** (monthly per-team PDF) — the backstop if "share" really means "report".

### Do **NOT** build
**Databricks App** (Sin can't deploy there; 24/7 billing; 2-4× cost; user-auth was Preview) · **OpenSharing/Delta Sharing** (structurally impossible on one metastore, and it can't share row-filtered tables) · **Legacy SQL Dashboard** (retired) · **Embedding** (that IS her link fear) · **Lakehouse Federation** (wrong tool entirely).

## 5. Addressing her security concern (C9) — evidence

**"If we export the link, anyone can access it" — FALSE.**
- **There is no public link in Databricks.** A viewer must be a **registered Entra/Databricks account identity** *and* be **explicitly granted CAN VIEW**.
- Authentication is **Entra SSO**; **no anonymous access** is supported.
- **IP access lists apply to all viewers**, including account users.
- **The one setting that could cause her fear:** *"Anyone in my account can view"*. **Do not use it** — share to **named groups only**.

**"AI/BI Dashboard is not secure" vs a Databricks App — head to head:**

| | AI/BI Dashboard (Individual data permissions) | Databricks App (user authorization) |
|---|---|---|
| Identity used against UC | Viewer | Viewer |
| RLS / column masks | Yes (UC) | Yes (UC) |
| Auth | Entra SSO, no anonymous | Entra SSO, no anonymous |
| Maturity | **GA** | user-auth was **Public Preview** |
| Custom code in the trust path | **None** | **Yes** — must forward `x-forwarded-access-token`; a bug **silently queries as the app's service principal = full data** |
| Ways to misconfigure into a leak | **1** (wrong publish mode) | **≥4** |

**Verdict:** the dashboard has a *smaller* misconfiguration surface than the App and is GA. **The legitimate kernel of her concern is real, though:** the **default publish mode ("Share data permissions")** runs as the publisher → **row filters evaluate as the publisher → every team sees every row.** That is the actual footgun. Name it, and show the control.

**Mitigations to offer formally:** Individual data permissions + UC row filter + named-group sharing (never "anyone in account") + SSO + IP access list + **SQL results download disabled** + audit logging (`system.access.audit`).

## 7. Watch-outs (these WILL bite)
1. **⚠️ Catalog isolation = the silent killer.** If `ISOLATED`, cross-workspace options fail — and **Option C fails invisibly** (account users get **blank widgets**, no error). Check **before** promising anything: `databricks workspace-bindings get-bindings catalog <cat>`.
2. **⚠️ Publish mode.** Default = **Share data permissions** = one identity for everyone = **all teams see all rows**. Put "choose *Individual*" in the written handover.
3. **⚠️ ACCOUNT groups, not workspace-local groups.** `is_account_group_member()` — **never `is_member()`**. `CREATE GROUP` in SQL makes workspace-local groups; don't.
4. **⚠️ `GRANT EXECUTE` on the row-filter function** — the most-forgotten grant.
5. **⚠️ Imported dashboards land in the importer's user folder** → move to `/Workspace/Shared/…` or nobody else sees them.
6. **⚠️ Change the `warehouse_id`** after import — otherwise it points at coredata's warehouse (IT pays / it breaks).
7. **⚠️ Subscriptions + RLS: TEST with 2 users from 2 teams** before shipping. If it blasts one identical PDF to everyone → fall back to one schedule per team.
8. **⚠️ Entitlement migration 2026-07-27 / 2026-09-14** — two weeks away.
9. **Row filter limits:** cannot be applied to a **VIEW**; not supported via Iceberg-REST / UC REST; no time travel; no clones; DBR ≥ 12.2 (dedicated access mode needs 15.4 LTS + serverless). **DBR < 12.2 "fails securely" = returns no data** — looks like a bug, not a permission error.
10. **A materialized view / streaming table built on the filtered table by a privileged owner bakes in unfiltered rows.** Never allow one without its own filter.
11. **Type mismatch in the row-filter UDF can silently return ALL rows** — match the parameter type to the column exactly and run with `spark.sql.ansi.enabled = true`.
12. **`.lvdash.json` format drifts** across releases — don't build a golden-file test on it.

## 9. Lock-in note
The **data** stays portable (Delta on ADLS). What you lock in is the **serving layer**: `.lvdash.json` is proprietary, AI/BI dashboards don't render outside Databricks, ABAC/row filters are UC-only, Genie is entirely proprietary. **The deliberate exit path is Option J** — same UC table + row filter, consumed by **Power BI over their own warehouse** — governance stays in UC, only the viz layer moves. Write it down so the choice is informed, not accidental.
