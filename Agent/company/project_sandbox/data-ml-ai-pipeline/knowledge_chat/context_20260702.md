## Key unblock this session: **CRD vs CR (instance)**
สิน was looking at a ~12,000-line YAML and was confused. Clarified:
- **CRD** (`kind: CustomResourceDefinition`, file `040-Crd-kafka.yaml`) = the **schema / "empty form"** that DEFINES what a `Kafka` resource may contain. It's **public Strimzi** (identical for everyone on that version) — **no need to screenshot it**; just name a field and it can be explained from public docs. It ships with the operator install.
- **CR instance** (`kind: Kafka`, a short ~30-100 line file with real values like `replicas: 3`) = the **actual cluster**. THIS is what creates a cluster. Still **unseen** — สิน should find it via Bitbucket search `kind: Kafka` (top-of-file, not `CustomResourceDefinition`), likely under `dev/` / `datazone/` / repo root of `dtp_kafka_cluster`.
- How to tell them apart: CRD = long, starts `kind: CustomResourceDefinition`, has `openAPIV3Schema`. Instance = short, starts `kind: Kafka`, has real values.

## Producer vs topic nuance (สิน asked)
- **connect = the source-DB / producer side** (Debezium reads Oracle/SQL Server → publishes events).
- **connect WRITES INTO topics** but does **not define/own** them. The topic **object lives in the broker cluster**; the topic **definition** (`kind: KafkaTopic`) lives in **dtp_kafka_cluster**.
- Open check: are topics **explicit** (`KafkaTopic` CRs present per topic) or **auto-created** (`auto.create.topics.enable=true`, named via connector `database.server.name`/`topic.prefix`)? Check how many `KafkaTopic` files exist in the cluster repo.
- Analogy used: source DB → [Connect/Debezium = delivery truck] → [topic = shelf in the warehouse=cluster] → [Databricks = truck taking goods out].
